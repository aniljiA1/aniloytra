TASK 01

Web App QA & Debug Report

Tested the Conduit RealWorld demo — a full-stack Angular + Node social blogging app. Exercised all primary flows: sign-up, login, create/edit/delete articles, add comments, follow users, edit profile, and logout. Found 5 confirmed issues ranging from Critical to Low severity.

Critical #1 — JWT Stored in localStorage (XSS Risk)

Steps: Open DevTools → Application → Local Storage after logging in. Observe jwtToken key with full bearer token value.

Expected: Token in HttpOnly, Secure cookie inaccessible to JS. Actual: Token readable by any script on the page — any XSS yields full account takeover.

Suspected cause: Auth service calls localStorage.setItem('jwtToken', token) directly — common in tutorial code, violates OWASP token storage guidance.

High #2 — No Rate-Limiting or CAPTCHA on Login

Steps: Submit wrong credentials 20+ times on the Sign In page. No lockout, no CAPTCHA, constant response time observed.

Expected: Account lockout or exponential back-off after N failures. Actual: Unlimited brute-force attempts accepted.

Suspected cause: /api/users/login has no rate-limit middleware (e.g. express-rate-limit); no bot protection on frontend.

High #3 — Article Body Accepts Unlimited Length

Steps: Paste 50,000+ characters into the article Body field → click Publish. Article publishes; very long unbreakable words break rendered page layout.

Expected: Character limit enforced with visible counter. Actual: No maxlength on textarea; backend accepts any payload size; Markdown renderer doesn't handle overflow.

Suspected cause: Missing input validation on both client and server; no payload size cap in API middleware.

Medium #4 — Generic 422 Error, No Field-Level Feedback

Steps: Sign Up with an already-taken username. Submit.

Expected: Inline error next to the username field. Actual: Flat error list appears at top of form ("username has already been taken") with no field association — easy to miss, especially on mobile.

Suspected cause: API 422 response errors are rendered as a list at form top rather than mapped to originating input fields.

Low #5 — Missing ARIA Labels on Tag Filter & Pagination

Steps: Run Lighthouse Accessibility audit on Home page, or navigate with VoiceOver/NVDA. Tag filter pills and Previous/Next pagination links have no text alternatives.

Expected: All interactive elements have descriptive aria-label. Actual: Pagination uses bare arrow characters; Lighthouse score ~61/100.

Suspected cause: Angular template components omit aria-label/aria-describedby; no a11y linting enforced in CI.

Issue #1 Deep-Dive: JWT in localStorage

What is happening?

When a user logs in, the Angular frontend receives a JWT from the backend and immediately writes it to window.localStorage under the key 'jwtToken'. This token is read back on every subsequent request and attached as an Authorization header. It persists across browser sessions until the user explicitly logs out.

Why is this a problem?

localStorage is accessible to any JavaScript running on the page — including scripts injected by a Cross-Site Scripting attack. If an attacker can inject even a single line of script (via a malicious article body, compromised CDN dependency, or browser extension), they can silently read and exfiltrate the token. Possessing the JWT is equivalent to being the user — the attacker can make authenticated API calls until the token expires.

Root cause

The Angular auth service calls localStorage.setItem('jwtToken', token) directly. The application also lacks a Content Security Policy header, Subresource Integrity on CDN scripts, and DOMPurify on Markdown output — any of these gaps could serve as the XSS vector.

How I would fix it

Store the token exclusively in an HttpOnly, Secure, SameSite=Strict cookie set by the server — JavaScript cannot read HttpOnly cookies, so XSS cannot steal the token. The frontend should stop reading or writing the JWT entirely; the browser attaches the cookie automatically. In parallel: add a strict CSP header, add SRI hashes to all CDN scripts, and sanitise Markdown output with DOMPurify.

APIEndpointPurpose
GitHub Search API GET /search/repositories Free, no key needed. Returns starred repos filtered by topic + creation date — ideal structured source for a digest.
GitHub Contents API GET /repos/{owner}/{repo}/readme Enrichment call — fetches the README of each top repo to add context beyond the name/description. Falls back gracefully to "(README unavailable)" on 404.
Discord Webhook POST {webhook_url} Free, zero-infra delivery. Embed format renders a clean digest card. Webhook URL stored in n8n Credentials — never hard-coded.

Threshold rationale (1,000 ⭐): Repos above 1k stars have demonstrated community validation and receive the full embed (description + README snippet). Below 1k are "early signal" — surfaced as a compact bullet list. The GitHub query is scoped to created:>2024-01-01 so even 200-star repos represent fast-rising new projects worth watching.

Error handling: Every HTTP node has continueOnFail: true with the error output wired to the Error Handler. If GitHub search fails, Discord gets a 🚨 alert. If a single README fetch fails, the Code node substitutes a placeholder so the rest of the digest still publishes. If Discord itself fails, n8n logs the execution failure in its audit trail.
